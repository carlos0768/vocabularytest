## 1. Project Overview

MERKEN (internal repo name: "wordsnap") is an AI-powered vocabulary learning PWA for Japanese English learners. Users photograph handwritten notes or printed materials, AI extracts English words with Japanese translations, and the app builds quizzes, flashcards, and sentence exercises from those words. The production domain is `https://www.merken.jp`.

The app is a Next.js web app that also has a companion iOS native app (separate codebase in `/mobile/`). The web app serves both as the primary product and as the payment gateway for iOS (KOMOJU) and handles IAP verification for the App Store path.

## 5. Major Data Flows

### Flow 1: Image Scan ŌåÆ Word Extraction (Web path)

1. User triggers scan on home page ŌĆö `src/app/page.tsx` (handles file input + mode selection)
2. `ScanModeModal` shown ŌĆö `src/components/home/ScanModeModal.tsx` (mode: all/circled/highlighted/eiken/idiom/wrong)
3. `ProcessingModal` shown ŌĆö `src/components/home/ProcessingModal.tsx` (step-by-step progress UI)
4. Image converted to base64 in-browser; HEIC converted via `heic2any` ŌĆö `src/lib/image-utils.ts`
5. `POST /api/extract` called ŌĆö `src/app/api/extract/route.ts`
   - Auth verified via Supabase (cookie or Bearer token)
   - Scan limit checked via Supabase RPC `check_and_increment_scan`
   - AI extraction called based on mode (see `getProvidersForMode`)
   - Gemini 2.5 Flash processes image ŌåÆ JSON words returned
6. Words stored in `sessionStorage` key `scanvocab_extracted_words` ŌĆö in-page JS
7. Navigation to `/scan/confirm` ŌĆö `src/app/scan/confirm/page.tsx`
8. User edits/confirms word list
9. On save: `getRepository(subscriptionStatus)` called ŌåÆ creates project + words in DB
10. Background: `prefillQuizData()` calls `/api/generate-quiz-distractors` (GPT-4o-mini generates 3 distractors + example sentence per word)
11. Pro only: `prefillQuiz2Data()` calls `/api/embeddings/sync` + `/api/quiz2/similar/batch`
12. Pro only: `prefillWordInsights()` calls `/api/generate-word-insights` (POS tags, related words, usage patterns)
13. Home page cache invalidated, redirect to home or project page

### Flow 2: Image Scan ŌåÆ Word Extraction (iOS/Async path via Scan Jobs)

1. iOS app uploads image to Supabase Storage directly
2. `POST /api/scan-jobs/create` called ŌĆö creates job record in `scan_jobs` table
3. `POST /api/scan-jobs/process` called (triggered by client polling or webhook) ŌĆö `src/app/api/scan-jobs/process/route.ts`
4. Same extraction logic runs server-side; word data saved to Supabase directly
5. Push notification sent via APNS (`src/lib/notifications/apns.ts`) or Web Push
6. Client downloads completed job and presents confirm screen

### Flow 3: Quiz

1. User navigates to `/quiz/[projectId]` ŌĆö `src/app/quiz/[projectId]/page.tsx`
2. Words loaded from repository (local or remote based on subscription)
3. Quiz questions built: each word + 3 distractors shuffled via `shuffleArray()`
4. If distractors missing, calls `/api/regenerate-distractors` (lazy generation)
5. Answer selection ŌåÆ `recordCorrectAnswer()` / `recordWrongAnswer()` updates localStorage stats
6. Word status updated via `repository.updateWord()` with SM-2 spaced repetition fields
7. Stats sync callback fires ŌåÆ `POST /api/activity` (daily log) + optional remote stats sync

### Flow 4: Subscription Activation (KOMOJU)

1. User visits `/subscription` ŌåÆ clicks upgrade ŌåÆ `POST /api/subscription/create`
2. KOMOJU session created (`mode: 'customer_payment'`) with `user_id` in metadata
3. User redirected to KOMOJU hosted payment page
4. Payment complete ŌåÆ KOMOJU POSTs to `POST /api/subscription/webhook`
5. Webhook handler verifies HMAC-SHA256 signature (timing-safe comparison)
6. Idempotency via `claim_webhook_event` Supabase RPC (prevents double-processing)
7. `activateBillingFromSession()` called ŌåÆ updates `subscriptions` table to `status='active'`, `plan='pro'`, `pro_source='billing'`
8. Client subscription cache invalidated on next page load

### Flow 5: Subscription Activation (Apple IAP)

1. iOS purchase completes ŌåÆ app sends `transactionId` to `POST /api/subscription/appstore/verify`
2. Server uses `@apple/app-store-server-library` to verify transaction with Apple
3. Checks `IAP_PRO_PRODUCT_IDS` whitelist
4. Updates `subscriptions` table: `pro_source='appstore'`, `current_period_end` from transaction

## 7. Danger Zones

### 7.1 Webhook Idempotency Logic
`src/app/api/subscription/webhook/route.ts` ŌĆö The webhook handler uses a Supabase RPC `claim_webhook_event` to prevent duplicate processing. If this RPC changes behavior or the `webhook_events` table schema changes, payments could double-activate or silently fail. The entire webhook is the critical payment activation path.

### 7.2 Subscription Status Computation
`src/lib/subscription/status.ts:getEffectiveSubscriptionStatus()` ŌĆö This function is called in multiple places (use-auth.ts, middleware, webhook handler, appstore verify). It has complex logic: active billing/appstore subscriptions expire when `currentPeriodEnd` passes; test pro expires via `testProExpiresAt`; `pro_source='none'` always returns 'cancelled'. If logic changes here without updating all callers consistently, Pro users could lose access or free users could gain Pro.

### 7.3 Hybrid Repository Full Sync
`src/lib/db/hybrid-repository.ts:fullSync()` ŌĆö Deletes all local data for a userId and replaces with remote. Has a safety guard (skip if remote is empty but local has data, line 108-113), but this is the most destructive operation in the codebase. Bug here = data loss for Pro users.

### 7.4 Scan Limit Enforcement
`src/app/api/extract/route.ts` ŌĆö Scan limits are enforced server-side via `check_and_increment_scan` Supabase RPC. The client also tracks locally in `localStorage` (via `src/lib/utils.ts:getDailyScanInfo()`), but the server check is authoritative. If the RPC behavior changes or the Pro check logic is altered, free users could bypass daily limits.

### 7.5 GlobalAuthState Singleton
`src/hooks/use-auth.ts` ŌĆö `globalAuthState`, `globalListeners`, `isGlobalLoading`, and `hasInitialized` are module-level variables (true singletons). This means all `useAuth()` callers share one state object. The optimistic load pattern reads Supabase session from localStorage before any async call. If this state gets corrupted (e.g., during account switching), all components will see stale data until a full page reload.

### 7.6 SessionStorage Dependency at /scan/confirm
`src/app/scan/confirm/page.tsx` ŌĆö Uses `sessionStorage` keys `scanvocab_extracted_words`, `scanvocab_project_name`, `scanvocab_project_icon`, `scanvocab_existing_project_id` to pass extracted words from the scan page. If these are missing (browser tab closed, session cleared), the page immediately redirects away. This is a fragile handoff point with no retry mechanism.

### 7.7 DB Schema: Divergence Between Migration File and Reality
`supabase/migrations/001_initial_schema.sql` shows the initial schema, but ~50 subsequent migration files have been applied. The CLAUDE.md schema section is outdated (only shows initial schema). The actual database has additional columns: `pro_source`, `test_pro_expires_at`, `cancel_at_period_end`, `cancel_requested_at`, `icon_image`, `share_id`, `is_favorite`, spaced repetition fields on words, embeddings, `subscription_sessions` table, `webhook_events` table, `scan_jobs` table, etc.

### 7.8 Cloud Run AI Proxy
When `CLOUD_RUN_URL` and `CLOUD_RUN_AUTH_TOKEN` are both set, all AI extraction bypasses direct API keys. The `CloudRunProvider` in `src/lib/ai/providers/cloud-run.ts` takes over. If Cloud Run is down, extraction fails entirely (no per-request fallback within the web app; fallback logic lives in Cloud Run itself).

### 7.9 Supabase Service Role Key Usage
`SUPABASE_SERVICE_ROLE_KEY` is used in: webhook handlers, scan-jobs process route, OTP routes, ops/api-costs route, AppStore verify route. This key bypasses RLS. Any bug in these routes that accepts attacker-controlled input could be catastrophic. The SQL injection guard (`scripts/check-sql-injection-guard.mjs`) runs in `npm run lint` as a defense.

## 14. Open Questions / Unresolved Risks

1. **RLS on newer tables**: The initial migration enables RLS on `subscriptions`, `projects`, `words`. There are ~40 subsequent migrations adding new tables. It is not confirmed that all new tables (`scan_jobs`, `webhook_events`, `subscription_sessions`, `collections`, `word_embeddings`, `web_push_subscriptions`, `ios_device_tokens`, `user_activity_logs`, etc.) have RLS enabled. The memory file references an incident with RLS on `subscriptions` (`incidents-rls.md`). This should be audited.

2. **`wasPro` flag in getRepository**: The `getRepository()` factory accepts `wasPro: boolean`. In `use-projects.ts:L19`, `wasPro` is computed as `subscription?.plan === 'pro' && subscriptionStatus !== 'active'`. However, in `scan/confirm/page.tsx:L499`, `getRepository(subscriptionStatus)` is called without `wasPro`, meaning downgraded users could get `localRepository` instead of `readonlyRemoteRepository` during save. This is a potential inconsistency.

3. **Free user word limit client-side only**: The 100-word limit for free users is enforced in `scan/confirm/page.tsx` client-side. There is no server-side enforcement of this limit at the word creation level. A free user could theoretically bypass this by calling the API directly.

4. **Grammar feature status**: The CLAUDE.md references a grammar learning feature (`/grammar/[projectId]/scan`, `/grammar/[projectId]`), but these routes were not found in the current `src/app/` directory tree. Either they were removed or not yet created. `vercel.json` references `src/app/api/grammar/route.ts` for timeout configuration, suggesting an API route may exist. Needs verification.

5. **`/api/security` endpoint**: There is an `src/app/api/security/` directory but its route file was not read. Its purpose and auth requirements are unknown.

6. **Admin route at `/admin`**: `src/app/admin/` exists with `[...slug]/page.tsx` and `page.tsx`. Auth requirements and purpose of this admin UI are not confirmed.

7. **`dictation` feature**: `/app/api/dictation/grade/route.ts` and `/app/dictation/page.tsx` exist. The dictation grading feature uses OpenAI but its integration into the main flow is not traced. Not mentioned in CLAUDE.md.

8. **`translate` API**: `/app/api/translate/route.ts` exists. Use cases and who calls this are unclear.

9. **`projects/` vs `project/[id]`**: Both `/app/projects/page.tsx` and `/app/project/[id]/page.tsx` exist. The top-level `projects` page may be an older all-projects list vs the individual project view. Relationship between these two routes is unclear.

10. **Cloud Run authentication model**: `CLOUD_RUN_AUTH_TOKEN` is a shared token. If it leaks, all AI extraction is compromised. The token is used as a bearer token. Rotation strategy is unclear.

11. **iOS push via APNS vs Web Push**: Both `src/lib/notifications/apns.ts` and `src/lib/notifications/web-push.ts` exist. How the app decides which to use per user is not fully traced.
