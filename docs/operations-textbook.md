## 1. まず全体像: Merken は何のサービスか

Merken は、**英単語学習に特化したAIサービス**です。

ユーザー体験を一言でいうとこうです。

1. 画像をアップロードする
2. AIが英単語を抽出する
3. 日本語訳やクイズ材料を作る
4. 単語帳として保存する
5. クイズや復習で学ぶ

つまり運営上は、Merken を次の5つの層に分けて考えると整理しやすいです。

### 1-1. 画面層
ユーザーが触るWeb画面。
- ホーム
- スキャン
- 単語帳
- クイズ
- 設定
- 課金ページ

### 1-2. API層
画面の裏側で動く「仕事の受付窓口」。
例:
- `/api/extract` = 画像から単語を取り出す
- `/api/subscription/...` = 課金処理
- `/api/ops/api-costs` = 管理用コスト確認
- `/api/scan-jobs/...` = 非同期スキャン処理

### 1-3. AI層
AIにお金がかかる部分。
主に以下。
- **Gemini**: 画像から単語抽出
- **OpenAI**: 誤答生成、例文生成、関連処理

### 1-4. データ層
保存先。
- **無料ユーザー**: 主にブラウザ内保存（IndexedDB）
- **Proユーザー**: Supabase にクラウド保存 + ローカルキャッシュ

### 1-5. 課金・認証・通知層
- Supabase Auth
- Stripe / App Store 課金
- Push通知

### 1-6. いちばん重要な運営理解
Merken は単なるWebサイトではなく、実質は以下の複合体です。

- **Next.js アプリ**
- **Supabase を使った会員・DBシステム**
- **Gemini / OpenAI を使うAIアプリ**
- **Stripe / Apple IAP を使う課金アプリ**

なので、運営者として最低限必要なのは「HTMLやCSSの知識」よりも、**どの機能がどの外部サービスに依存しているか**を理解することです。

## 2. 運営に必要な分野一覧

日常運営で必要な知識を、重要度順に並べるとこうなります。

### 最重要A: サービス全体の経路理解
「どの操作で、どのAPIが呼ばれ、どこに保存され、どこで課金が発生するか」を理解すること。

これがないと、
- コスト削減
- 不具合切り分け
- 機能削減判断
- 新機能の採算判断

ができません。

### 最重要B: AIコスト構造
Merken は AI を使うので、**普通のWebアプリより変動費が読みづらい**です。

特に理解が必要なのは:
- どの機能が Gemini を使うか
- どの機能が OpenAI を使うか
- その呼び出しは必須か、後回し可能か
- 失敗しても本体機能が生きるか

### 最重要C: 保存の仕組み
「無料ユーザーはローカル保存、Proユーザーはクラウド保存」が重要。

これを理解していないと、
- データ消失の意味
- Pro価値の意味
- 同期バグの重さ

を見誤ります。

### 重要D: 課金・会員状態
- 誰が Pro か
- 何で課金されるか
- 課金失敗時にどう見えるか

これは売上に直結します。

### 重要E: 障害対応の基本
- どこが壊れると何が止まるか
- どのログを見るか
- 応急処置と根本修正をどう分けるか

### 中重要F: デプロイ・変更管理
- 何を変えたら公開されるか
- main に入ると何が起こるか
- 本番反映前に何を確認するか

### 後回しG: SQL・低レベル実装
必要になる場面はあるが、日常運営の最低限ではありません。

## 3. システム全体の地図

## 3-1. ユーザー価値の中心動線

Merken の中核動線はこれです。

**画像アップロード → 単語抽出 → 保存 → 学習**

運営判断では、まずこの中核動線を最優先で守ります。

理由:
- ここが壊れると「Merkenらしさ」が消える
- 課金以前に継続率が落ちる
- 他機能より価値の核に近い

### 3-2. 主なAPIの役割

#### `/api/extract`
最重要API。画像から単語を抽出する。

役割:
- 認証確認
- スキャン制限確認
- Gemini で抽出
- 日本語補完
- 例文の補助生成

**壊れると:** サービスの顔が死ぬ

#### `/api/scan-jobs/*`
iOS / 非同期系スキャン処理。

**壊れると:** 一部のスキャンが終わらない、詰まる

#### `/api/subscription/*`
課金系。

**壊れると:** 売上に直結、Proの有効化が止まる

#### `/api/ops/api-costs`
管理者用コスト確認。

**壊れると:** サービス提供自体は続くが、経営判断が鈍る

### 3-3. 外部サービスの依存関係

- **Vercel**: サイト本体の公開
- **Supabase**: 認証、DB、クラウド保存
- **Gemini**: 画像から単語抽出
- **OpenAI**: クイズ補助、例文、埋め込み系
- **Stripe / Apple IAP**: 課金
- **Cloud Run**: AI呼び出しの中継・運用補助（構成次第）

運営者として重要なのは、障害時に「どこが落ちたのか」をこの依存関係から逆算できることです。

## 4. コスト構造の教科書

Merken のコストは大きく4種類あります。

1. **ホスティング費**（Vercel）
2. **DB / 認証 / ストレージ費**（Supabase）
3. **AI API費**（Gemini / OpenAI）
4. **決済まわりの手数料**（Stripe / Apple IAP）

この中で、最も運営判断が必要なのは通常 **AI API費** です。

## 4-1. なぜAIコスト理解が最重要なのか

VercelやSupabaseはある程度固定寄りですが、AIコストは**利用量と設計で大きく変わる**からです。

同じユーザー数でも、
- 1回のスキャンで何回AIを呼ぶか
- 例文生成を即時でやるか後回しにするか
- Proだけにするか全員に開放するか

で利益率が変わります。

## 4-2. AIコストを考えるときの基本フレーム

機能ごとに以下を確認します。

1. **どのモデルを使うか**
2. **いつ呼ばれるか**
3. **1回で何回呼ばれるか**
4. **それは必須か、補助か**
5. **無料ユーザーにも出すか、Pro限定か**

この5つでほぼ判断できます。

## 4-3. Merken のAI機能を分類するとこうなる

### A. コアAI（削ると価値が崩れる）
- 画像から単語抽出

これはサービスの核なので、**最後まで守る対象**です。

### B. 価値増幅AI（便利だが、設計次第で調整可能）
- 例文生成
- 誤答選択肢生成
- 類似語や関連情報生成

これはコスト最適化の主戦場です。

### C. 贅沢AI（状況次第で削減候補になる）
- 高頻度の再生成
- 重い補助機能
- 利用率が低いのにAIを多く使う機能

## 4-4. コスト削減の考え方

コスト削減では、いきなりモデルを安いものに変える前に、次の順で考えるのが基本です。

### 優先順位1: 呼び出し回数を減らす
一番効きます。

例:
- 保存時に毎回例文生成するのをやめる
- 使われない補助AIを後回しにする
- 同じ結果を再利用する

### 優先順位2: 即時処理を非同期化する
- ユーザーの待ち時間を減らす
- 重い処理を後ろに回せる
- 失敗しても本体体験を守りやすい

### 優先順位3: 無料→Proで線を引く
- AIコストが高い機能は Pro 限定に寄せる
- 「使う人だけがコストを発生させる」設計にする

### 優先順位4: モデルやプロンプトを軽量化する
- ただし品質低下とトレードオフ

## 4-5. 削減してよいもの / だめなもの

### 削減候補になりやすい
- 補助例文
- 再生成系
- 類似語・埋め込みなど、体験の本丸ではないもの
- 使用率の低い高度機能

### 基本的に最後まで守る
- 画像から単語抽出
- 保存
- ログイン
- 課金有効化

## 5. 保存構造の教科書

Merken の保存構造は、運営判断上かなり重要です。

## 5-1. 無料とProで保存先が違う

### 無料ユーザー
- 主にブラウザ内保存（IndexedDB）

意味:
- サーバーコストが安い
- ただし端末依存が強い
- ブラウザや端末事情で消えるリスクがある

### Proユーザー
- Supabase にクラウド保存
- ローカルキャッシュ併用

意味:
- 同期価値がある
- 複数端末利用に強い
- Proの価値の中核になる

## 5-2. 運営者が理解すべきこと

保存の問題は、単なる技術問題ではなく**プラン価値そのもの**です。

つまり:
- 同期が壊れる = Pro価値が壊れる
- ローカル保存問題 = 無料体験の信用が落ちる
- データロス = 一番致命的な信用毀損

## 5-3. 保存まわりで重要な問い

不具合報告を受けたら、まず次を聞く/考える。

1. 無料ユーザーか Proユーザーか
2. 同じ端末だけで使っていたか
3. ログインしていたか
4. データが「消えた」のか「見えなくなった」のか
5. Webか iOSか

この切り分けだけで、かなり原因候補が絞れます。

## 6. 日常運営で見るべきポイント

毎日・毎週見るべき観点を整理するとこうです。

## 6-1. 毎日見るべきもの

### 1. スキャンが正常に動いているか
最優先。

見る理由:
- 中核体験だから
- AI障害が出やすいから
- ユーザー離脱に直結するから

### 2. 課金が正常に有効化されているか
- 決済成功なのにPro反映されない事故は最悪

### 3. 大きなエラーや失敗通知がないか
- Vercelデプロイ失敗
- webhook失敗
- scan job 詰まり

### 4. AIコストの急増がないか
- 新機能公開後は特に注視

## 6-2. 毎週見るべきもの

### 1. どの機能が使われているか
使われない高コスト機能は削減候補。

### 2. 無料→Proの導線が機能しているか
- Pro価値が見えているか
- 無料機能を出しすぎていないか
- 逆に無料体験が弱すぎないか

### 3. AI費用対効果
- そのAI呼び出しは継続率や売上に効いているか
- 「すごいけど使われてない」機能は危険

## 7. 障害対応の教科書

障害時に最初にやるべきことは、**原因特定ではなく被害範囲の把握**です。

## 7-1. まず分類する

障害はだいたい次のどれかです。

### A. 公開障害
- サイトが開かない
- 画面が真っ白

主に Vercel / ビルド / 本番コードの問題。

### B. 認証障害
- ログインできない
- セッションが切れる

主に Supabase Auth / middleware / 環境変数。

### C. スキャン障害
- 画像アップロード後に止まる
- 単語抽出されない

主に `/api/extract`、Gemini、Cloud Run、scan jobs。

### D. 保存障害
- 保存できない
- データが消えた
- 同期されない

主に Supabase / repository / local-remote 差分。

### E. 課金障害
- 支払ったのにProにならない
- サブスク状態が変

主に webhook / subscription logic。

## 7-2. 見る順番

1. **どのユーザーに影響しているか**
   - 全員か、一部か
2. **どのプラットフォームか**
   - WebかiOSか
3. **どの機能で止まったか**
   - ログイン、抽出、保存、課金
4. **外部依存のどこが怪しいか**
   - Vercel / Supabase / Gemini / OpenAI / Stripe

## 7-3. 重要原則

### 原則1: 中核導線を先に守る
まず守るのは:
- ログイン
- スキャン
- 保存
- 課金反映

### 原則2: 補助AIは止めてもよい
例文や補助生成は、必要なら一時停止してでも本体を守る。

### 原則3: 原因究明と応急処置を分ける
- 応急処置 = ユーザー影響を早く止める
- 根本修正 = 後でやる

## 8. デプロイと変更管理の教科書

## 8-1. 公開とは何か

Merken における「公開」は、単にコードを置くことではなく、次の全体が成立して初めて公開です。

- コードが本番に反映される
- 必要な環境変数がある
- DBスキーマが合っている
- 外部APIが正常
- 課金や認証が壊れていない

## 8-2. 変更の危険度

### 低危険
- テキスト修正
- 軽いUI調整
- 表示順変更

### 中危険
- スキャン導線変更
- 保存フロー変更
- 認証画面変更

### 高危険
- 課金ロジック変更
- subscription status 変更
- scan job 処理変更
- repository / sync ロジック変更
- migration を伴う変更

## 8-3. 公開前に最低限見るべきこと

1. スキャンできるか
2. 保存できるか
3. ログインできるか
4. Pro判定が壊れていないか
5. 重大なコスト増になる処理を入れていないか

## 9. 課金・会員状態の教科書

運営上、課金は「支払い」だけでなく、**権限管理**でもあります。

## 9-1. 運営で理解すべき問い

- そのユーザーは本当に Pro 扱いか
- 何経由のProか（Stripe / App Store / テスト付与など）
- 課金したのに反映されていないのか
- 反映はされているが画面表示だけ変なのか

## 9-2. 課金障害の危険性

課金障害は2種類あります。

### 1. 売上を失う障害
払えない、課金導線が壊れている。

### 2. 信用を失う障害
払ったのに Pro にならない。

後者のほうが評判ダメージが大きいこともあります。

## 9-3. 課金まわりで覚えるべき最低限

- 課金成功 = それだけでは不十分
- **Webhookや検証処理を通って、初めて状態反映**される
- だから「決済会社では成功してるのに、アプリでは失敗」というズレが起こりうる

## 10. 運営者の意思決定フレーム

新機能・改善・削減判断は、次の5軸で見ると整理しやすいです。

## 10-1. 5つの判断軸

### 1. 中核価値への近さ
Merkenらしさに近いか。

### 2. 利用頻度
本当に使われるか。

### 3. AIコスト
継続的に重いか。

### 4. 壊れた時の被害
止まると何が死ぬか。

### 5. Pro価値への寄与
有料化の理由になるか。

## 10-2. 典型判断

### 残すべき機能
- 中核価値に近い
- よく使われる
- 多少コストがかかっても継続率に効く

### 削るべき候補
- 利用率が低い
- AIコストが高い
- 壊れやすい
- 差別化に効いていない

### Pro限定に寄せるべき機能
- コストが高い
- 一部ユーザーしか使わない
- でも刺さる人には価値が高い

## 15. まとめ

Merken 運営で最初に必要なのは、技術を細かく書けることではありません。

必要なのは、

- **このサービスの価値の核は何か**
- **どこでお金がかかるか**
- **どこが壊れると何が止まるか**
- **何は削ってよく、何は守るべきか**

を判断できることです。

運営者としての最低限の知識は、
**SQLの暗記ではなく、構造の理解**です。

その意味で、Merken の運営知識は次の一文に要約できます。

> スキャン・保存・学習・課金の流れと、その裏で動く AI / DB / 認証 / 課金の依存関係を理解することが、運営の土台である。

---

## 付録A. 最低限の用語集

- **API**: 画面の裏で動く処理の窓口
- **Vercel**: Webアプリの公開先
- **Supabase**: DB・認証・保存の基盤
- **IndexedDB**: ブラウザ内ローカル保存
- **Webhook**: 外部サービスからの自動通知
- **RLS**: ユーザーごとにDBアクセスを制限する仕組み
- **Cloud Run**: 一部バックエンド処理の実行先
- **Scan Job**: 非同期スキャン処理単位
- **Repository Pattern**: 保存先の違いを吸収する設計
- **Pro gating**: 有料会員だけに機能を開く制御

---

## 付録B. 運営者向け超短縮版

もし3分で思い出したいなら、これだけでいいです。

1. Merken の核は **スキャン → 保存 → 学習**
2. 一番重要な外部依存は **AI / Supabase / 課金**
3. 一番見ないといけないのは **AIコストと中核導線の正常性**
4. 削る候補は **補助AI**、守るべきは **抽出・保存・課金**
5. 運営者に必要なのは実装暗記ではなく **構造理解**
