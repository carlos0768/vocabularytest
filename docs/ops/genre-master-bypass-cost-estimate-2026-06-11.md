# ジャンル設定ユーザのマスター例文バイパス コスト予測 (2026-06-11)

PR #178（ジャンル設定ユーザはマスター例文を読み込まず毎回生成＋ジャンル強制紐づけプロンプト）の
コスト影響の試算。`docs/ops/example-genres-cost-estimate-2026-06-10.md` の
「再試算が必要になる条件」に該当する仕様変更のため、本書で再試算する。

## 変更内容（コストに関係する部分）

1. **マスター例文の読み込みスキップ**: ジャンル設定ユーザのスキャンでは、
   マスター（`lexicon_entries`）に例文がある単語でも prefill せず、
   全単語をAI例文生成の対象にする（`skipMasterExamples`）。
   → **AI呼び出し回数そのものが増える**（前回試算との最大の違い）。
2. **ジャンル指示ブロックの強化**: 指示文が約135文字（2項目）から
   約230文字（見出し＋4項目）に増加。
   → ジャンル設定ユーザの生成コール1回あたり入力トークンが微増。

ジャンル未設定ユーザは両方とも対象外のため **コスト増ゼロ**（従来挙動のまま）。

## 結論（サマリ）

| 項目 | 影響 |
|------|------|
| AI呼び出し**回数** | **増える**。ジャンル設定ユーザのスキャンで、従来マスター例文を流用していた語も1語1コールで生成（例文生成は1語ずつ個別コール・5並列） |
| AI**入力トークン** | 上記の新規コール分＋既存コールへの指示文増分（+70〜100トークン/コール） |
| AI**出力トークン** | 新規コール分のみ増（約120トークン/語） |
| 直接コスト増 | **月あたり数百〜1万円規模**（下記モデル参照）。前回試算（数十〜数百円）より約1桁大きい |
| レイテンシ | ジャンル設定ユーザのスキャン処理が **+数秒〜10秒程度** 延びる（同期生成のため） |
| 二次的影響 | ほぼ中立（後述） |
| DBコスト | 実質変化なし |

## 前提

- 例文生成はすべて **GPT-4o-mini**（`AI_CONFIG.defaults.openai`）。
  単価（試算時点・要再確認）: 入力 $0.15 / 1Mトークン、出力 $0.60 / 1Mトークン、$1 = 150円換算。
- 例文生成は **1語1コール**（`generateExampleSentences`、バッチ抜け回避のため）。
  1語あたり入力 約500トークン + 出力 約120トークン ≒ **0.022円/語**（前回試算と同じ）。
- 強化後のジャンル指示ブロックは約230文字 ≒ **170〜250トークン**
  （従来比 +70〜100トークン。コスト換算 +0.001〜0.002円/語で実質無視できる）。
- スキャン1回あたりの語数を約20語、うちマスター例文ヒットを約10語と仮定
  （マスターヒット率が上がるほどこの数は増え、本試算の増分も増える点に注意）。

## 1. 直接コスト（AI呼び出し回数増）

### スキャン1回あたり（ジャンル設定ユーザ）

| | 変更前 | 変更後 | 増分 |
|---|------:|------:|-----:|
| 例文生成コール数/スキャン | マスター未ヒット語のみ（約10語） | 全語（約20語） | **+約10語** |
| 例文生成コスト/スキャン | 約0.22円 | 約0.44円 | **+約0.22円** |

### 月間モデル

追加コスト ≒ 月間スキャン数 × ジャンル設定ユーザ比率 × マスター例文ヒット語数/スキャン × 0.022円

| 月間スキャン | ジャンル設定率 | マスター例文ヒット語数/スキャン | 追加コスト/月 |
|------------:|------------:|------------------:|-------------:|
| 10,000 | 30% | 10語 | 約 **660円** |
| 50,000 | 30% | 10語 | 約 **3,300円** |
| 100,000 | 50% | 10語 | 約 **11,000円** |

前回試算（プロンプト追記のみ・100〜1,700円/月）に対して **約1桁の増加**。
前回ドキュメントが予告していた通りのオーダー。

### 指示文増分（既存コール分）

ジャンル設定ユーザの全例文生成コールに +70〜100トークン。
月間10万語生成でも約 $1〜1.5 ≒ **150〜220円/月**。無視できる水準。

### 上限による歯止め

- 無料ユーザ: スキャン3回/日（サーバ強制）→ 最悪でも
  3スキャン × 20語 × 0.022円 ≒ **約1.3円/日/人**。
- `/api/generate-examples`（オンデマンド生成）は
  `AI_LIMIT_EXAMPLES_FREE_DAILY=15` / `AI_LIMIT_EXAMPLES_PRO_DAILY=150` の
  日次上限が引き続き効く。
- スキャン時生成（`/api/extract`, `/api/scan-jobs/process`）には語数ベースの
  上限がない点は従来通り。Proユーザのスキャン無制限 × ジャンル設定の
  組み合わせが実質的なコストドライバー。

## 2. レイテンシ・コンピュート

- スキャン時の例文生成は**同期実行**（5並列・1語数秒）。
  マスター例文ヒット10語が生成対象に加わると、2バッチ分 ≒ **+4〜10秒/スキャン**。
- それに伴い Vercel / Cloud Run の実行時間も同程度増加。
  AI料金に比べれば小さいが、UX（スキャン待ち時間）への影響の方が顕在化しやすい。

## 3. 二次的影響（マスター辞書まわり）

- ジャンル設定ユーザはマスター例文を**読みも書きもしなくなる**ため、
  マスター辞書の成長・他ユーザのヒット率への影響は **ほぼ中立**
  （即時書き戻しスキップは前回から、読み込みスキップが今回追加）。
- 夜間バックフィルcron（`20260324130000_nightly_example_backfill.sql`）は
  `words.example_sentence` をジャンル設定の有無に関係なく空のマスターへコピーする。
  ジャンル特化例文が空のマスターエントリに入り得るのは**従来からの既知挙動**で、
  今回の変更では増えない（マスターヒット語は既に例文を持つため `IS NULL` ガードで上書きされない）。

## 4. DB

- `/api/extract` で `fetchExampleGenres()` の呼び出しタイミングを前倒ししたが、
  回数は実質変わらず（全語が例文済みのケースで+1 SELECTになる程度）。
- `/api/scan-jobs/process` は既存の取得を流用しており増加なし。

## モニタリング

- 管理ダッシュボード `/api/ops/api-costs`（`api_cost_events`）で OpenAI
  入出力トークンの推移を確認。**デプロイ後1〜2週間は例文生成コストの伸びを重点監視**。
- スキャンログの `exampleGeneration` summary（生成語数）と
  `masterHitCount` の比率で、増分が想定（ヒット語数≒追加生成数）通りか確認。
- スキャン処理時間（`timing.exampleGenerationMs`）の p95 を確認し、
  ジャンル設定ユーザのスキャン体感が許容範囲か見る。

## 再試算が必要になる条件

- 例文生成モデルを GPT-4o-mini から変更したとき（`src/lib/ai/config.ts`）
- 例文生成を1語1コールからバッチに戻したとき
- スキャンあたりの平均語数・マスターヒット率が想定（20語・50%）から大きく
  ずれていることが計測で判明したとき
- ジャンル設定ユーザ比率が50%を大きく超えたとき
- スキャン時例文生成に語数上限・キャッシュ（ユーザ別例文の再利用）を導入したとき
